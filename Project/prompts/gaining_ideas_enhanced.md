# 产品设计提示词（增强版）—— AI 智能体专用

基于 `gaining_ideas.md` 的原始需求，本文件对提供给 AI 智能体的产品设计提示词进行了系统性完善。新增内容涵盖硬件规格、软件栈、模态定义、评估标准、输出格式等原提示词中缺失的关键信息。

---

## 1. 项目背景

在暑期学校中，Deep Learning 小组（3 人）与 Robotics 小组（2 人）共同完成一个机器人小车项目。使用树莓派和小车，运用 Deep Learning 和 Robotics 的知识设计并实现一个具有简单应用价值的作品。

**产品设计周期**：11 天（含设计、实现、调试、演示）

## 2. 硬件资源清单（原提示词缺失，此为新增关键信息）

### 2.1 计算平台

| 组件 | 型号/规格 | 说明 |
|------|----------|------|
| 主控板 | Raspberry Pi 5 | 作为深度学习推理和上层控制的计算平台 |
| 微控制器 | Arduino Mega 2560 | 负责底层电机控制、传感器读取、实时响应 |
| 通信方式 | USB 串口 | Arduino 通过串口接收树莓派指令 |

### 2.2 传感器（部分，如有另外需要可增加）

| 传感器 | 数量 | 说明 |
|--------|------|------|
| USB 摄像头 | 1 个 | 分辨率 ≥ 640×480，用于所有视觉任务 |
| 超声波传感器（HC-SR04） | 2-3 个 | 测距范围 2cm-4m，用于避障和距离控制 |
| 轮式编码器 | 4 个 | 追踪车轮转数，用于里程计 |

### 2.3 执行器（部分

| 执行器 | 数量 | 说明 |
|--------|------|------|
| 直流电机 + 驱动板（L298N） | 4 个 | 差速驱动，实现前进/后退/转向 |
| LED 灯（RGB 可编程） | 2-3 个 | 状态指示、交互反馈 |
| 蜂鸣器 | 1 个 | 声音提示 |
| OLED 屏幕（单色, 128×64） | 1 个 | 显示简单文字/状态信息 |

### 2.4 结构

| 组件 | 说明 |
|------|------|
| 车体 | 小型差速轮式底盘（手掌至 A4 纸大小） |
| 扩展材料 | 乐高结构件、3D 打印件（需预留打印时间） |
| 电源 | 充电电池组（5V/12V 双路输出） |

### 2.5 硬件局限性（明确约束）

1. **定位精度低**：仅靠轮式编码器追踪轨迹，无法进行长距离精确运动或严格循迹；无外部辅助定位设施（如 UWB、运动捕捉系统）
2. **载重受限**：小车规格较小，不能增加过多配重
3. **外设受限**：无激光雷达、深度相机、IMU、GPS 等高级传感器
4. **算力有限**：树莓派运行深度学习模型的能力有限，复杂模型可能无法实时推理
5. **摄像头固定**：摄像头高度和视角受小车尺寸限制（通常距地面 10-20cm）

## 3. 软件资源与环境

### 3.1 可用框架与库

| 类别 | 工具 | 用途 |
|------|------|------|
| 深度学习推理 | PyTorch / ONNX Runtime / TensorFlow Lite | 在树莓派上运行训练好的模型 |
| 计算机视觉 | OpenCV | 图像采集、预处理、ArUco 码检测、颜色识别 |
| 目标检测 | YOLOv8 / YOLOv11 (nano/small) | 实时目标检测（举手、人脸、物体等） |
| 图像分类 | ConvNeXt / ResNet / MobileNet | 物体分类（如手卡图案识别） |
| 网络通信 | Python socket / gRPC / MQTT | 树莓派与 Arduino 之间的通信 |
| Arduino 编程 | Arduino IDE / PlatformIO | 底层电机控制与传感器读取 |

### 3.2 推理性能约束

| 模型规模 | 树莓派推理速度（估算） | 适用场景 |
|---------|----------------------|---------|
| MobileNet-v3 | ~100-200ms/帧 | 轻量分类任务 |
| YOLOv8n (640×480) | ~200-500ms/帧 | 目标检测（需权衡帧率与精度） |
| ConvNeXt-Tiny | ~300-600ms/帧 | 精细分类任务 |
| 传统 CV（ArUco/颜色） | <5ms/帧 | 标记识别，不受模型推理速度限制 |

> **关键约束**：若单帧推理时间 >300ms，实时交互体验会显著下降。复杂模型需考虑云端推理（网络延迟约 50-100ms）的分布式方案。

## 4. 核心需求（原文提炼 + 补充解释）

### 需求 1：场景落地潜力
给出若干在**实际生活、教育等场景**下具有落地潜力的实现方向。创意应回答：
- **谁**是目标用户？
- 解决**什么**痛点？
- 在**什么环境**中使用？
- 为什么用机器人小车（而非手机 App / 固定设备）来解决？

### 需求 2：至少使用两种模态

**模态定义**（原提示词未定义，此处明确）：

在此项目中，"模态"指传感器/信息输入的**不同类型**，包括但不限于：

| 模态 | 对应硬件 | 可识别的信息 |
|------|---------|------------|
| 视觉（RGB 图像） | USB 摄像头 | 物体检测、人脸、手势、文字/标记、颜色 |
| 距离/深度 | 超声波传感器 | 障碍物距离、物体接近/远离 |
| 音频 | 麦克风（Pi 板载/USB） | 语音指令、拍手、特定声音 |
| 触觉/按钮 | 按钮/触摸传感器 | 用户主动触发 |
| 运动/里程 | 编码器 | 运动距离、速度 |
| 光强 | 光敏电阻 | 环境光变化、手部遮挡信号 |

> 最少 "两种模态" 的意图：避免仅依赖单一摄像头做所有事。模态组合应体现**互补性**（如视觉识别"谁"和"什么"，距离传感器确定"多远"和"安全停止"）。

### 需求 3：产品必须有可活动部件

- "可活动部件"定义：电机驱动的、在运行时会发生物理移动的部件
  - ✅ 满足：**车轮**（小车移动到不同位置）、舵机驱动的机械结构（夹爪、推杆、旋转台）
  - ✅ 满足：小车整体位置的移动本身就是活动部件
  - ❌ 不满足：仅 LED 闪烁、仅屏幕显示文字
- 产品本身可以固定在一处（即不要求小车必须自主移动），但意味着：
  - 若小车不移动，则需要**车上的机械部件**有活动（如机械臂、旋转平台等）
  - 若小车移动（巡游），则车轮本身就是活动部件

### 需求 4：基于 Deep Learning 的自动化特征

"自动化"在此语境下的层次（由低到高）：

| 层次 | 示例 | 自动化程度 |
|------|------|-----------|
| 感知自动化 | 自动检测举手、自动识别人脸 | ★★☆ |
| 决策自动化 | 根据检测结果自动决定小车行为（转向、停车） | ★★★ |
| 闭环自动化 | 感知→决策→执行→反馈，全流程无人干预 | ★★★★★ |

**最低要求**：至少达到"感知自动化"层。**推荐目标**：实现"决策自动化"层——即深度学习模型的输出直接驱动硬件行为，而非仅做信息展示。

### 需求 5：实时交互能力

"实时"在此项目中的操作化定义：

| 指标 | 可接受范围 | 理想范围 |
|------|----------|---------|
| 用户触发到小车响应 | < 3 秒 | < 1 秒 |
| 视觉感知延迟（含推理） | < 500ms | < 200ms |
| 交互循环频率 | ≥ 1 次交互 / 5 秒 | 持续流式交互 |

"交互"的形式可以是：
- 人主动触发 → 小车响应（如举手 → 小车靠近）
- 小车主动检测 → 小车做出行为 → 人做出回应
- 持续的双向互动（如小车跟随人、躲避障碍物）

|---------|---------|---------|------|------|
| 示例：运动控制模块 | 直流电机×2, 驱动板, 编码器 | 差速驱动、任意角度转弯、巡游 | 目标速度/角度 | 实际移动 |
| ... | ... | ... | ... | ... |

### 5.3 深度学习算法需求

对每个深度学习任务，描述：

| 任务 | 模型类型 | 输入 | 输出 | 推理平台 | 延迟要求 |
|------|---------|------|------|---------|---------|

### 5.4 模态融合方案

说明多种模态如何协作：

### 5.5 交互流程

绘制用户与小车交互的完整时间线（状态机或时序图）：
1. 初始状态
2. 触发条件
3. 感知阶段
4. 决策阶段
5. 执行阶段
6. 反馈阶段
7. 回到初始状态（或进入新状态）

### 5.6 可行性评估

| 维度 | 评分（1-5） | 说明 |
|------|-----------|------|
| DL 模型可行性 | | 模型能否在 11 天内完成训练/适配？推理速度是否满足实时要求？ |
| 硬件可行性 | | 现有硬件是否足够？是否需要额外采购（需预留到货时间）？ |
| 机械可行性 | | 机械结构能否用乐高/简单 3D 打印实现？ |
| 时间可行性 | | 按 11 天分解任务，是否可完成？ |
| 鲁棒性 | | 对环境变化（光照、噪音、人流）的容忍度？ |
| 演示效果 | | 最终演示时是否直观、有冲击力？ |

### 5.7 风险与备选方案
- 每个模块的最大技术风险是什么？
- 如果该模块在实现中遇到不可克服的困难，有什么"降级替代"方案？
- 最低可接受的产品形态（MVP）是什么？

## 7. 创意生成方法论（帮助 AI 更系统地产生想法）

请按以下步骤产出创意：

### Step 1：枚举模态组合
从视觉、距离、音频、触觉、运动五种模态中，选 2-3 个做组合，列出所有可能的模态对。

### Step 2：枚举交互触发方式
- 用户主动触发（举手、拍手、按键、语音指令、展示手卡）
- 环境自动触发（检测到特定物体、到达特定位置、特定时间）
- 持续交互（跟随、躲避、搜索）

### Step 3：枚举核心动作
- 移动类：靠近、远离、巡游、跟随、定点
- 操作类：抓取、推、拉、倾倒、旋转
- 信号类：亮灯、蜂鸣、显示文字

### Step 4：组合生成创意
将 Step 1-3 中的元素交叉组合，筛选出有意义、可实现的方案。

### Step 5：按可行性排序
用 §5.6 的可行性评估维度对每个创意打分，按综合得分排序，提供前 3-5 个最优方案。

## 8. 已知参考案例（帮助理解期望产出水平）

以下为本项目已有的一个详细设计方案（教室巡游举手响应小车），供理解设计的深度和详细程度：

**场景**：教室内的学生举手 → 小车巡游中发现举手 → 靠近 → 识别需求手卡 → 递送资源 → 回到巡游路线

**核心模块**：
- 运动模块：差速驱动 + 地面巡游线循迹 + 超声波避障
- 视觉模块：YOLOv8 举手检测 + ArUco 码手卡识别 + 帧压缩上传
- 通信模块：树莓派 ↔ Arduino 串口/WiFi + 云端 GPU 推理（可选）
- 交互模块：LED 状态指示 + 蜂鸣器提示 + 外置篮筐

**模态融合**：视觉（YOLO 找谁、ArUco 识别要什么）+ 距离（超声波安全停车）+ 里程（回归巡游线）

**深度学习任务**：举手检测（YOLOv8, 目标检测）+ 手卡识别（ArUco 码, 传统 CV 替代方案）

## 9. 约束条件速查表（设计自检清单）

- [ ] 是否有可活动部件（车轮移动 或 机械结构）？
