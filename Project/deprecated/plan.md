## 详细分析

### 1. 视频帧截取方案与频率

**架构定位**：根据 `prompt.md` 的分析，系统采用"树莓派（端）→ 网络通信 → 云端 GPU"的分布式架构。Arduino 小车通过摄像头采集视频流，树莓派作为边缘计算节点负责帧提取、压缩和转发。

**帧截取方式**：

在树莓派上使用 **OpenCV (`cv2.VideoCapture`)** 是最直接且成熟的方案：

```python
import cv2
cap = cv2.VideoCapture(0)  # USB摄像头通常为/dev/video0
cap.set(cv2.CAP_PROP_FRAME_WIDTH, 640)
cap.set(cv2.CAP_PROP_FRAME_HEIGHT, 480)
ret, frame = cap.read()  # 逐帧读取
```

关键参数设置：
- **分辨率**：建议设为 **640×480**（VGA）。理由：(a) 对于举手检测，640×480 分辨率已足够 YOLOv8 提取有效特征；(b) 高分辨率（如 1080p）会成倍增加 JPEG 压缩时间和网络传输量；(c) 低分辨率（如 320×240）可能丢失远处举手的小目标。
- **帧率设置**：摄像头硬件帧率可设为 30fps，但实际截取帧率应该远低于此。

**合适的截取频率**：

核心约束来自两个环节的延时：

| 延时环节 | 典型耗时 |
|---------|---------|
| 帧截取 + JPEG 压缩（树莓派） | 15-30ms |
| 网络传输（WiFi 校园网） | 10-50ms |
| 云端 YOLOv8 推理（举手检测） | 30-80ms (YOLOv8m/l on GPU) |
| 结果回传 + 小车控制响应 | 10-30ms |
| **单帧总延时** | **约 65-190ms** |

考虑到：
1. **prompt.md 明确指出**："如果视频流延迟超过 300ms，小车很容易发生控制震荡或开过头"
2. 教室内巡游场景，举手检测不需要极高实时性（学生在巡游路线上停留举手，而非高速移动目标）
3. 需要为偶尔的网络抖动留出缓冲

**推荐截取频率：3-5 FPS（即每 200-333ms 截取一帧）**

具体建议：
- **巡游阶段（举手检测）**：**3 FPS**（每 333ms 一帧）。举手是一个持续数秒的动作，3fps 意味着学生举手 2 秒内至少能被捕获 6 帧，足够触发"连续 10 帧检测到举手"的确认机制（约 3.3 秒确认）。
- **近距离交互阶段（手卡识别）**：**5 FPS**（每 200ms 一帧）。此时小车已停止运动，对延迟容忍度更高，可以略微提高帧率以获得更快的识别结果。

**避免网络拥塞的关键措施**：
- 采用 **漏桶/令牌桶速率控制**，严格限制每秒发送帧数
- 使用 JPEG 压缩质量参数（quality=60-70）大幅减小每帧体积
- 当网络延迟超过阈值时自动降帧率到 1-2 FPS

### 2. 图片压缩算法选择

对于"树莓派→网络→云端"的视频帧传输场景，压缩方案的选择需要权衡三个维度：**压缩比、压缩速度、图像质量（对检测精度的影响）**。

| 方案 | 压缩比 | 编码速度（树莓派） | 质量保持 | 适用性 |
|------|--------|-------------------|---------|--------|
| **JPEG** | 10:1~20:1 | 极快（<5ms @ 640×480） | 良好 | ★★★★★ |
| **WebP** | 15:1~25:1 | 中等（~15ms） | 良好 | ★★★★ |
| **PNG** | 2:1~5:1 | 快 | 无损 | ★★（体积过大） |
| **H.264 视频编码** | 极高 | 较慢（硬件编码除外） | 良好 | ★★★ |

**推荐方案：JPEG 压缩（quality=60~75）**

理由：
1. **压缩速度**：JPEG 是树莓派 CPU 上最快的图像压缩算法。OpenCV 的 `cv2.imencode('.jpg', frame, [cv2.IMWRITE_JPEG_QUALITY, 70])` 在树莓派 4B 上处理 640×480 帧仅需约 **3-5ms**，几乎不增加端侧延时。
2. **体积控制**：640×480 的原始 BGR 帧约 900KB，经 JPEG quality=70 压缩后约 **25-50KB**，压缩比达 18:1~36:1，大幅降低网络传输压力。
3. **检测精度影响**：YOLOv8 模型本身对 JPEG 压缩伪影有一定鲁棒性（训练数据常用 JPEG 格式）。quality≥60 时，举手检测和 ArUco 码识别的精度几乎不受影响。
4. **实现简单**：OpenCV 原生支持，无需额外依赖。

**具体参数建议**：
```python
encode_param = [int(cv2.IMWRITE_JPEG_QUALITY), 70]
result, encimg = cv2.imencode('.jpg', frame, encode_param)
# encimg 直接通过 socket 发送，无需 base64 编码
```

如果树莓派支持硬件编码（如使用 `picamera2` + 硬件 JPEG 编码器），可以进一步将编码时间压缩到 **<1ms**。

### 3. 网络通信方案选择

根据 `prompt.md` 中已有的分析，系统需要两种通信通道：

| 通道 | 方向 | 数据量 | 延迟要求 | 可靠性要求 |
|------|------|--------|---------|-----------|
| 视频帧上传 | 树莓派→云端 | 中（25-50KB/帧） | 低延迟 | 允许少量丢帧 |
| 推理结果下传 | 云端→树莓派 | 极小（几十字节） | 极低延迟 | 必须可靠 |
| 心跳/控制 | 双向 | 极小 | 中等 | 必须可靠 |

**推荐方案：双通道混合架构**

#### 通道一：视频帧传输 — **UDP + 自定义协议**

使用 UDP 传输 JPEG 压缩后的视频帧：
- **优势**：无连接建立开销、无拥塞控制的人为延迟、允许丢帧（符合"偶尔丢一帧不影响检测"的特性）
- **协议设计**：
  ```
  [FrameID: 4字节][TotalChunks: 2字节][ChunkIndex: 2字节][DataLength: 4字节][JPEG Data]
  ```
  - 每帧拆分为多个 chunk（每个 chunk ≤ 8KB），避免 IP 分片
  - FrameID 递增，云端可检测丢帧
  - 640×480 JPEG（约 30KB）约需 4 个 chunk

- **为什么不用 TCP 传图片**：TCP 的拥塞控制和重传机制会引入不可预测的延迟。当网络出现一次丢包，TCP 会暂停后续所有数据直到该包重传成功，这可能导致 200-500ms 的额外延迟——远超 300ms 的控制安全阈值。

#### 通道二：推理结果 + 心跳 — **TCP Socket / gRPC**

使用 TCP 传输推理结果和控制指令：
- 数据量极小（bounding box 坐标 + 类别 + 置信度，不到 100 字节）
- 必须可靠传输（丢了一个坐标可能让小车错过目标）
- TCP 的可靠性保证了"指令必达"

**更优选择：gRPC**
- 使用 Protocol Buffers 定义消息格式：
  ```protobuf
  message InferenceResult {
    int32 frame_id = 1;
    repeated Detection detections = 2;
    int64 timestamp_ms = 3;
  }
  message Detection {
    string class_name = 1;   // "raised_hand", "card_water", etc.
    float confidence = 2;
    int32 x_center = 3;
    int32 y_center = 4;
    int32 width = 5;
    int32 height = 6;
  }
  ```
- gRPC 基于 HTTP/2，支持双向流、自带心跳保活
- 序列化后的消息体积极小

#### 网络穿透方案

`prompt.md` 中提到的 **FRP 隧道**是必要的：
- 云端 GPU 服务器部署在校园网之外（可能是有公网 IP 的云服务器或内网工作站）
- 树莓派在教室局域网内，通过 FRP 客户端连接公网 VPS 中继
- FRP 配置：将树莓派上的 gRPC 端口和 UDP 端口都映射出去

#### 整体通信架构图

```
[树莓派/Arduino小车]
    │
    ├── UDP (port 9001) ──→ JPEG 视频帧 ──→ [云端 GPU 服务器]
    │                                         │
    ├── TCP/gRPC (port 9002) ←── 推理结果 ──┤
    │                                         │
    ├── TCP/gRPC (port 9002) ──→ 心跳包 ──→  │
```

- 树莓派每 **200ms** 发送一次心跳包

- 这对应了 `prompt.md` 中强调的"弱网断连回退机制 (Fallback)"

### 问题 1：步骤 3 中的误判阈值设为多少比较合适？

**核心矛盾**：需要平衡"快速响应"与"避免误触发"。阈值太低会导致误判（学生短暂抬手整理头发就被判定为举手），阈值太高会导致响应迟钝。

**分析维度**：

| 因素 | 影响 |
|------|------|
| 截取帧率（3 FPS） | 每帧间隔约 333ms |
| 举手动作持续时间 | 正常举手动作持续 2-5 秒 |
| 误触发场景 | 伸懒腰、整理头发（通常 < 1.5 秒） |
| YOLOv8 单帧检测精度 | 远距离场景存在间歇性漏检/误检 |
| 小车转向控制 | 确认后需立即响应，延迟不宜过大 |

**建议阈值：连续 9-12 帧（即 3-4 秒内持续检测到举手）**

具体推荐 **连续 10 帧**，理由如下：

1. **10 帧 × 333ms = 约 3.3 秒确认时间**。这足以过滤掉绝大多数偶然性抬手动作（伸懒腰、整理头发等通常不超过 2 秒），同时 3.3 秒的响应延迟对于"学生举手→小车转向"的交互场景是可接受的。

2. **容错设计**：不应要求"严格连续 10 帧"。建议采用 **滑动窗口 + 容错机制**：
   - 在过去 N 帧（如 15 帧 ≈ 5 秒）的滑动窗口中，如果有 **≥ 10 帧检测到举手**（且 bounding box 中心点 x 坐标偏差不超过帧宽的 15%，即"相近位置"），即判定为真正举手
   - 这样允许 1-2 帧的间歇性漏检（远距离 YOLOv8 可能偶尔漏检），不会因一两帧的检测失败就重置计数

3. **距离适配**：若小车距离目标较远（通过 YOLO 检测框面积估算），可以适当放宽阈值（如 8 帧即可确认），因为远处目标的检测本身就不稳定；近距离时（框面积大）可收紧到 10-12 帧。

**推荐的伪逻辑**：
```
if 滑动窗口(最近15帧)中举手帧数 >= 10 AND 举手位置标准差 < 帧宽15%:
    → 确认举手，锁定目标
```

### 问题 2：通过物体在图中的几个代表性点位与实际角度确定映射关系的方法

这个问题的本质是：**已知物体在图像中的水平像素位置（x 坐标），求解物体相对于摄像头的真实方位角（水平偏转角）**。

**业内常用方法：基于相机标定的多项式拟合 / 查表法**

#### 2.1 理论基础

在针孔相机模型下，水平方位角 θ 与图像 x 坐标的关系为：

$$
\theta = \arctan\left(\frac{x - c_x}{f_x}\right)
$$

其中 $c_x$ 是光心 x 坐标（图像中心），$f_x$ 是水平焦距（像素单位）。

但实际摄像头存在**镜头畸变**（尤其是广角镜头），且摄像头安装可能不在精确的旋转中心，导致上述理想公式偏差较大。

#### 2.2 推荐方案：多点标定 + 多项式拟合

**标定流程**：
1. 将小车置于旋转台（或地面标记旋转刻度），摄像头安装在小车旋转中心
2. 在固定距离（如 2 米）放置举手者/标定物
3. 旋转小车（或标定物移动），每 5°-10° 记录一组数据点：
   - 实际方位角 θ（已知，通过地面刻度读取）
   - 标定物中心在图像中的水平像素坐标 x
4. 收集约 15-30 组数据点（覆盖 -60° 到 +60° 的视野范围）

**拟合方法**：
- **3 次多项式拟合**（最常用）：$\theta = a_3 x^3 + a_2 x^2 + a_1 x + a_0$
  - 优点：能较好地拟合镜头畸变带来的非线性，计算量极小
  - 工具：NumPy 的 `np.polyfit(x_data, theta_data, 3)`
  
- **分段线性插值 / 查表法**（更精确但需要更多标定点）：
  - 将 x 坐标划分为密集区间，每个区间内线性插值
  - 优点：不需要假设函数形式，精度取决于标定密度
  - 适合对精度要求极高的场景

**关键注意事项**：
- 标定时**标定物距离必须固定**（如 2m），因为同样 x 偏移在不同距离下对应不同角度
- 如果摄像头不在旋转中心，还需要额外的**偏移修正项**
- 建议验证拟合残差，确保最大误差 < 2°，否则需要增加标定点或提高多项式阶数

### 问题 3：超声波 vs YOLO 图像识别，哪种方案控制小车靠近举手者最合适？

**结论：推荐 YOLO + 超声波融合方案，以 YOLO 为主、超声波为安全兜底。**

| 维度 | 超声波 | YOLO 图像识别 |
|------|--------|--------------|
| 距离测量精度 | 高（厘米级） | 低（通过 bbox 面积间接估算） |
| 目标区分能力 | **无**（只能检测最近障碍物） | **强**（能区分举手者与其他物体/人） |
| 环境适应性 | 受多路径反射、吸音材料影响 | 受光线变化、遮挡影响 |
| 近距离（<0.5m） | 精度高，盲区小 | 举手者可能超出视野 |
| 中距离（0.5-3m） | 无法区分目标 | 能持续跟踪目标 |
| 实现复杂度 | 极低（传感器读数） | 需要模型推理 |

#### 推荐的三阶段靠近策略：

**阶段一：远程接近（>2m）—— 纯 YOLO 引导**
- YOLO 持续检测举手者的 bounding box
- 通过问题 2 中的标定映射，将 bbox 中心 x 坐标转换为方位角，控制小车转向
- 前进速度恒定（低速），仅调整方向

**阶段二：中距离接近（0.5m - 2m）—— YOLO + 超声波粗检**
- YOLO 继续提供转向引导
- 超声波开始工作，测量前方最近障碍物距离
- 此时超声波读数应与 YOLO 估算距离**交叉验证**：若两者差异过大（>50cm），说明 YOLO 可能跟踪了错误目标或超声波检测到了其他障碍物 → 停车、重新扫描
- 通过 bbox 面积增长速率估算接近速度，避免冲撞

**阶段三：最终靠近（<0.5m）—— 超声波为主、YOLO 为辅**
- 近距离时举手者可能超出摄像头视野（尤其是广角不够时），YOLO 可能丢失目标
- **超声波接管距离控制**：以 30-50cm 为目标停止距离，超声波精确控制最后一段
- YOLO 仅在能检测到时提供身份确认（确保面前还是举手者）
- 超声波设定安全阈值：距离 < 20cm → 紧急停车

#### 为什么不是纯超声波方案？

超声波最大的问题是**无法区分目标**。在教室场景中，小车前方可能有多个人、桌椅等障碍物。超声波只能返回"最近物体距离"，无法判断这个最近物体是否是举手者。如果非举手者站在小车和举手者之间，纯超声波方案会错误地停在该人面前。

#### 为什么不是纯 YOLO 方案？

YOLO 估算距离的精度很差——通过 bbox 面积反推距离依赖于对"举手者实际大小"的假设，不同学生的体型差异会导致数米级别的误差。近距离时，目标可能超出视野或严重变形，YOLO 的检测不稳定。

**最终建议**：YOLO 负责"找谁"和"往哪走"（身份识别 + 方向引导），超声波负责"停在哪"（精确距离控制），两者互补。

| 问题 | 推荐方案 |
|------|---------|
| 误判阈值 | 滑动窗口 15 帧内 ≥10 帧检测到举手 + 位置偏差 < 15%帧宽 |
| 映射关系 | 多点标定 + 3 次多项式拟合 $\theta = f(x)$ |

### 问题 1：步骤 4 中，采用什么方式来识别手卡上的图案难度较低且较为用户友好？

**推荐方案：ArUco 码 / AprilTag（二维码式标记）**

**难度极低**：

- 无需记住特定手势或图案，纸板上可以同时用文字标注"我要水"辅助人类理解。
- ArUco 码支持**任意旋转角度识别**，用户不必刻意对齐方向。

- 需要采集不同光线、角度、学生手绘风格的训练数据，标注工作量大。
- 远距离下自定义图案分辨率极低，YOLOv8 难以准确区分。

**具体实现建议**：

- 近距离交互时，挂起 YOLOv8 举手检测模型，切换到 ArUco 检测。
- 设定 10 秒超时，未识别到则自动退出，回到巡游状态。

### 问题 2：树莓派可以如何通过网络信道下传识别结果给小车上的 Arduino 主板？可以采用什么协议进行网络通信？

**核心认知**：根据 `prompt.md` 的分析，Arduino 传统型号（Uno、Mega2560）**没有原生 Wi-Fi 能力和足够的 RAM 来处理网络协议栈**。因此有两种硬件拓扑，决定了不同的通信方式：

#### 拓扑 A：Arduino 通过串口连接树莓派（树莓派在车上）

如果树莓派也在小车上（通过 USB 线连接到 Arduino），这是**最简单、最可靠**的方案：

- **物理层**：USB 串口（`/dev/ttyUSB0` 或 `/dev/ttyACM0`）
- **协议**：**自定义轻量串口协议**
- **通信格式**：
  ```
  [指令类型: 1字节][参数1: 2字节][参数2: 2字节][校验和: 1字节]\n
  ```
  示例：
  - `0x01` = 前进, 参数为速度
  - `0x02` = 转向, 参数为角度
  - `0x03` = 停车
  - `0x10` = 识别结果, 参数为资源编码（0=水, 1=笔, 255=无法识别）
  - `0x11` = 蜂鸣器提示
  - `0x12` = 灯光报警

- **优势**：零延迟、零丢包、无需网络配置、不受 WiFi 质量影响。

#### 拓扑 B：Arduino 通过 ESP8266/ESP32 接收 WiFi 信号（树莓派不在车上）

如果树莓派部署在远端（教室角落或机房），小车需要无线接收指令：

- **物理层**：WiFi (2.4GHz)
- **传输层协议**：**TCP Socket**（简单可靠）
- **应用层协议选项**：
  1. **TCP + 自定义二进制协议**（最轻量）：与串口协议相同，通过 TCP Socket 发送原始字节流。Arduino 端用 ESP8266/ESP32 的 WiFi 库接收。
  2. **TCP + JSON**（更易调试）：`{"cmd":"TURN","angle":45,"speed":80}`，Arduino 端用 ArduinoJson 库解析。
  3. **MQTT**（适合多设备场景）：树莓派发布到 `car/control` 主题，ESP32 订阅该主题。优点是自带 QoS 和断线重连。

- **端口设计**：树莓派作为 TCP Server，监听端口 8888；小车上的 ESP32 作为 TCP Client 连接。

#### 推荐方案：拓扑 A（串口）+ 拓扑 B（TCP）混合

考虑到 `prompt.md` 中明确说"树莓派提供算力，不在小车上"，实际采用的是**拓扑 B**。推荐使用 **TCP + JSON** 协议：

- **为什么用 TCP 而不是 UDP**：识别结果数据量极小（几十字节），但**必须可靠送达**。丢了一个识别结果意味着用户的需求被忽略，这是不可接受的。TCP 的重传机制保证了"指令必达"。
- **为什么用 JSON 而不是二进制协议**：10 天开发周期内，JSON 的可调试性远优于二进制协议（可直接用 `nc` 或 Python 脚本测试），序列化开销对几十字节的消息完全可以忽略。
- **心跳机制**：小车上的 ESP32 每 500ms 发送 `{"type":"heartbeat"}` 心跳，树莓派超过 2 秒未收到心跳则判定连接断开。

## 详细分析：问题 4 — 巡游方式与抗干扰策略

### 一、问题核心拆解

这个问题的本质是两个子问题：
1. **采用什么方式让小车能忍受外部干扰（PID 偏差、外力位移）仍按预定路线行进？**
2. **小车从与需求者交互到回到巡游状态的运动特点是什么？据此选择最恰当的巡游方式。**

---

### 二、小车运动的完整状态机分析

根据 plan.md 的任务拆解，小车在一个完整周期中经历以下状态：

```
巡游 → 检测举手 → 转向靠近 → 停车识别手卡 → 响应需求 → 逆向返回 → 恢复巡游
```

**关键运动特点**：

| 阶段 | 运动特点 | 位置不确定性 |
|------|---------|------------|
| **巡游** | 沿预设路线匀速前进 | 低（循迹） |
| **转向靠近** | 偏离巡游路线，朝举手者前进，记录转角 θ 和前进距离 d | 逐渐增大 |
| **停车识别** | 静止，但与需求者交互时可能被意外推动 | 可能跳变 |
| **逆向返回** | 反向行驶（-θ 角度，-d 距离）试图回到脱轨点 | **累积误差最大** |
| **恢复巡游** | 从"理论上回到的点"继续沿原路线巡游 | **关键：若未精确回到原位，将偏离巡游线** |

**核心矛盾**：plan.md 第 27 行明确指出——**"PID 控制不一定能保证小车精确回到步骤 3 中转向之前的原点"**。逆向返回存在以下误差源：
- 电机转速不一致导致的转向角度误差（±2-5°）
- 轮子打滑导致的前进距离误差（±5-15%）
- 外力推动导致的姿态漂移
- 这些误差在多次交互后**累积放大**，最终小车完全偏离预定路线

---

### 三、巡游方式方案对比

#### 方案 A：纯里程计 + PID 死推算（Odometry-based）

小车通过编码器记录里程，PID 控制速度和转向，完全依赖内部状态估计位置。

| 维度 | 评价 |
|------|------|
| 实现难度 | ★★☆☆☆ |
| 抗干扰能力 | ★☆☆☆☆（极差） |
| 成本 | 零额外成本 |
| 长期稳定性 | ★☆☆☆☆（误差累积无界） |

**致命缺陷**：无法感知外部位移。如果小车被人推动 10cm，里程计完全不知道这件事，后续所有位置估计全部偏移。

---

#### 方案 B：地面粘贴巡游线 + 红外/颜色传感器循迹（Line Following）

在地面粘贴黑色/彩色胶带作为巡游路线，小车底部安装红外或颜色传感器沿线路前进。

| 维度 | 评价 |
|------|------|
| 实现难度 | ★★☆☆☆ |
| 抗干扰能力 | ★★★★☆（偏离后能自动纠正回线） |
| 成本 | 低（一卷黑色胶带 + 红外传感器模块） |
| 长期稳定性 | ★★★★★ |

**优势分析**：
1. **闭环纠正**：无论小车因何种原因偏离路线，传感器检测到"离线"后会立即转向纠正，形成**负反馈闭环**
2. **回归巡游的优雅性**：交互结束后逆向返回，不需要精确回到脱轨点——只要大致回到巡游线附近，传感器会自动引导小车"骑上"巡游线，然后沿线路继续前进
3. **10 天开发周期友好**：红外循迹是 Arduino 最经典的入门项目之一，代码和硬件调试成熟

**对岔路口（日型走廊）的处理**：

这是 plan.md 第 26 行提出的关键问题。日型走廊需要小车在岔路口做出路线选择。解决方案：

- **方案 B1：巡游线编码**（推荐）
  - 在岔路口前的巡游线上设置"编码标记"（如并排双线、不同颜色标记）
  - 传感器检测到编码标记后，小车根据预设路线表选择左转/右转/直行
  - 例如：主路线 = 黑色单线；岔路口前 5cm 处出现红色标记 = "下一岔口右转"

- **方案 B2：预设路线序列**
  - 巡游路线是固定的闭环（如"日"字形的外圈 → 内圈 → 外圈）
  - 小车维护一个路口计数器：经过第 1 个路口右转，第 2 个路口直行，第 3 个路口左转...
  - 每次回到巡游线后计数器不重置，继续按序列走

---

#### 方案 C：视觉 SLAM / 地标定位（Visual Landmark）

在教室天花板或墙壁粘贴 ArUco 码作为全局定位标记，小车摄像头同时用于巡游定位。

| 维度 | 评价 |
|------|------|
| 实现难度 | ★★★★☆ |
| 抗干扰能力 | ★★★★★ |
| 成本 | 需要额外打印 ArUco 码 |
| 10 天可行性 | ★★☆☆☆（风险高） |

**问题**：
- 小车摄像头在巡游时需要**同时完成举手检测和定位**两个任务，算力和帧率冲突
- 需要标定多个地标码的全局坐标，前期工作量大
- 从 ArUco 码估计位姿需要较好的视角，低角度小车摄像头可能无法稳定看到天花板标记

---

#### 方案 D：超声波墙壁跟随（Wall Following）

小车侧面安装超声波传感器，保持与教室墙壁的固定距离，沿墙壁巡游。

| 维度 | 评价 |
|------|------|
| 实现难度 | ★★★☆☆ |
| 抗干扰能力 | ★★★☆☆ |
| 适用性 | ★★☆☆☆（教室内部路线可能不贴墙） |

**问题**：教室内的巡游路线不一定沿着墙壁走（如穿过课桌之间的走道），墙壁跟随适用场景有限。

---

### 四、最终推荐：方案 B（巡游线循迹）— 基于运动特点的必然选择

**推荐理由总结**：

#### 1. 从运动特点看：回归巡游是最薄弱环节

小车在交互结束后逆向返回，存在**里程计累积误差 + 外力扰动 + 轮子打滑**三重误差。方案 B 的巡游线提供了**绝对位置参考**——小车不需要精确回到脱轨点，只需要回到巡游线**附近**（例如 ±3cm 以内），红外传感器就能自动将小车拉回线上。

这正是**"容忍外部干扰"的工程答案**：不是让 PID 更精确，而是引入一个**绝对参考系**，把开环控制变成闭环控制。

#### 2. 从 10 天开发周期看：方案 B 风险最低

- 红外循迹模块（TCRT5000 或类似）成本不到 ¥10，Arduino 库和教程极其丰富
- 代码量约 50-100 行，调试时间约 1-2 天
- 地面粘贴黑色胶带（电工胶布），教室地面通常为浅色，对比度足够

#### 3. 从系统鲁棒性看：方案 B 天然支持中断恢复

无论小车在何时何地被外力推动、被抱起再放下、被意外阻挡——只要它还在巡游线附近，传感器就能让它重新"骑上"巡游线。这是纯里程计方案永远做不到的。

### 五、推荐的具体实现方案

#### 5.1 硬件配置

```
[车体底部前部]
├── TCRT5000 × 3（左、中、右，间距约 1.5cm）
│   ├── 左传感器 → 检测是否偏右
│   ├── 中传感器 → 检测是否在线上
│   └── 右传感器 → 检测是否偏左
└── 黑色电工胶带（宽 1.5-2cm）贴于地面形成巡游路线
```

#### 5.2 控制逻辑（PID 循迹）

```
if 中传感器=黑(线上) → 直行
elif 左传感器=黑 → 小车偏右了，左转修正
elif 右传感器=黑 → 小车偏左了，右转修正
else → 三传感器全白 → 丢失巡游线 → 原地旋转搜索
```

#### 5.3 岔路口处理

对于日型走廊，采用**路口序列法**：

```python
# 预设巡游路线序列
ROUTE = ["直行", "右转", "直行", "左转", "直行", "右转", ...]
crossing_count = 0  # 路口计数器

# 三传感器同时检测到黑线 = 岔路口
if left == BLACK and center == BLACK and right == BLACK:
    action = ROUTE[crossing_count % len(ROUTE)]
    execute(action)  # 执行对应转向
    crossing_count += 1
```

#### 5.4 与举手检测的协调

```
主循环（Arduino 端）:
  while True:
    if 收到举手信号:
      暂停巡游 → 执行转向靠近 → 交互 → 逆向返回
      恢复巡游（从当前路口计数器继续）
    else:
      巡游循迹（红外传感器 + PID 控制）
      每 333ms 发送一次摄像头帧给树莓派
```
